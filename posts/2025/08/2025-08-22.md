Designing a well-structured REST API is crucial for building maintainable and scalable applications. Following established conventions ensures your API is intuitive, predictable, and easy to use.

## REST Principles

REST (Representational State Transfer) is an architectural style for designing web services:

- **Stateless**: Each request contains all information needed
- **Cacheable**: Responses should be cacheable when possible
- **Uniform Interface**: Consistent resource identification and manipulation
- **Client-Server**: Separation of concerns
- **Layered System**: Can have multiple layers (proxies, gateways)

## Resource Naming

### Use Nouns, Not Verbs

```http
# ❌ Bad
GET /getUsers
POST /createUser
DELETE /deleteUser

# ✅ Good
GET /users
POST /users
DELETE /users/:id
```

### Use Plural Nouns

```http
# ✅ Consistent plural naming
GET /users
GET /posts
GET /comments
```

### Hierarchical Resources

```http
# Nested resources
GET /users/:userId/posts
GET /users/:userId/posts/:postId/comments

# Alternative flat structure
GET /posts?userId=123
GET /comments?postId=456
```

## HTTP Methods

Use appropriate HTTP methods:

| Method | Purpose | Example |
|--------|---------|---------|
| GET | Retrieve | `GET /users` |
| POST | Create | `POST /users` |
| PUT | Replace | `PUT /users/:id` |
| PATCH | Partial update | `PATCH /users/:id` |
| DELETE | Remove | `DELETE /users/:id` |

## Status Codes

Return meaningful status codes:

### Success

```http
200 OK         # Successful GET, PUT, PATCH
201 Created    # Successful POST
204 No Content # Successful DELETE
```

### Client Error

```http
400 Bad Request      # Invalid request
401 Unauthorized     # Authentication required
403 Forbidden        # Insufficient permissions
404 Not Found        # Resource doesn't exist
409 Conflict         # Resource conflict
422 Unprocessable    # Validation error
```

### Server Error

```http
500 Internal Server Error # Server error
502 Bad Gateway           # Invalid response from upstream
503 Service Unavailable    # Temporarily unavailable
```

## Request/Response Format

### Consistent Structure

```json
// Request
POST /users
Content-Type: application/json

{
  "name": "John Doe",
  "email": "john@example.com"
}

// Success Response
HTTP/1.1 201 Created
Content-Type: application/json

{
  "data": {
    "id": 123,
    "name": "John Doe",
    "email": "john@example.com",
    "createdAt": "2025-08-22T10:00:00Z"
  }
}

// Error Response
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid email format",
    "details": {
      "field": "email",
      "reason": "Must be a valid email address"
    }
  }
}
```

## Pagination

Provide pagination for list endpoints:

```http
GET /users?page=1&limit=20

Response:
{
  "data": [...],
  "pagination": {
    "page": 1,
    "limit": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

### Cursor-Based Pagination

```http
GET /users?cursor=abc123&limit=20

Response:
{
  "data": [...],
  "pagination": {
    "nextCursor": "def456",
    "hasMore": true
  }
}
```

## Filtering and Sorting

```http
# Filtering
GET /users?status=active&role=admin

# Sorting
GET /users?sort=createdAt&order=desc

# Combining
GET /posts?authorId=123&sort=title&order=asc&limit=10
```

## Versioning

Version your API:

```http
# URL versioning
GET /v1/users
GET /v2/users

# Header versioning
GET /users
Accept: application/vnd.api+json;version=1
```

## Authentication

Use standards-based authentication:

```http
# Bearer token
GET /users
Authorization: Bearer <token>

# API key
GET /users
X-API-Key: <key>
```

## Rate Limiting

Implement rate limiting:

```http
# Headers
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1234567890

# 429 Too Many Requests
HTTP/1.1 429 Too Many Requests
Retry-After: 60
```

## Documentation

Document your API thoroughly:

```yaml
# OpenAPI/Swagger example
openapi: 3.0.0
info:
  title: User API
  version: 1.0.0
paths:
  /users:
    get:
      summary: List users
      parameters:
        - name: page
          in: query
          schema:
            type: integer
      responses:
        '200':
          description: Successful response
```

## Best Practices

1. **Consistent naming** - Use nouns, plural form
2. **Proper status codes** - Use standard HTTP codes
3. **Versioning** - Plan for API evolution
4. **Pagination** - Handle large datasets
5. **Filtering** - Allow flexible queries
6. **Error handling** - Clear error messages
7. **Security** - Authentication and authorization
8. **Rate limiting** - Protect your API
9. **Documentation** - Clear API docs
10. **Testing** - Comprehensive tests

## Common Mistakes

- Using verbs in URLs
- Inconsistent status codes
- Not versioning APIs
- Missing pagination
- Poor error messages
- No rate limiting
- Overfetching data
- Underfetching data

Following REST principles and best practices creates APIs that are intuitive, maintainable, and scalable.
