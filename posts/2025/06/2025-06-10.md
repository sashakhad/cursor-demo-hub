Accessibility is not optional—it's a fundamental requirement for creating inclusive web experiences. Building accessible applications ensures everyone can use your products, regardless of abilities.

## Why Accessibility Matters

Web accessibility benefits everyone:
- **Legal compliance**: Meets WCAG and legal requirements
- **Broader audience**: Includes users with disabilities
- **Better UX**: Clear, predictable interfaces
- **SEO benefits**: Semantic HTML improves search rankings
- **Mobile optimization**: Often improves mobile experience

## Semantic HTML

Use semantic elements for their intended purpose:

```html
<!-- ❌ Bad -->
<div onclick="handleClick()">Submit</div>

<!-- ✅ Good -->
<button type="submit">Submit</button>
```

### Good Structure

```html
<header>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
    </ul>
  </nav>
</header>

<main>
  <article>
    <h1>Article Title</h1>
    <p>Content...</p>
  </article>
</main>

<footer>
  <p>&copy; 2025 My Company</p>
</footer>
```

## ARIA Labels

Use ARIA attributes when semantic HTML isn't enough:

```html
<!-- Icon buttons -->
<button aria-label="Close dialog">
  <span aria-hidden="true">×</span>
</button>

<!-- Form fields -->
<label for="email">Email</label>
<input 
  id="email" 
  type="email" 
  aria-required="true"
  aria-describedby="email-error"
/>
<span id="email-error" role="alert">
  Please enter a valid email
</span>

<!-- Landmarks -->
<nav aria-label="Breadcrumb">
  <ol>
    <li><a href="/">Home</a></li>
    <li aria-current="page">Current Page</li>
  </ol>
</nav>
```

## Keyboard Navigation

Ensure all interactive elements are keyboard accessible:

```typescript
// Focus management
function Modal({ isOpen, onClose }: ModalProps) {
  const modalRef = useRef<HTMLDivElement>(null);
  
  useEffect(() => {
    if (isOpen) {
      // Focus modal when opened
      modalRef.current?.focus();
      
      // Trap focus inside modal
      const firstFocusable = modalRef.current?.querySelector(
        'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
      );
      firstFocusable?.focus();
    }
  }, [isOpen]);
  
  return (
    <div
      ref={modalRef}
      role="dialog"
      aria-modal="true"
      aria-labelledby="modal-title"
      tabIndex={-1}
    >
      <h2 id="modal-title">Modal Title</h2>
      <button onClick={onClose}>Close</button>
    </div>
  );
}
```

## Color Contrast

Ensure sufficient contrast ratios:

```css
/* ✅ Good contrast */
.high-contrast {
  color: #000000; /* Text */
  background: #ffffff; /* Background */
  /* Contrast ratio: 21:1 */
}

/* ❌ Poor contrast */
.low-contrast {
  color: #aaaaaa;
  background: #ffffff;
  /* Contrast ratio: 2.3:1 - fails WCAG AA */
}
```

Don't rely on color alone:

```html
<!-- ❌ Bad -->
<span class="error">Required field</span>

<!-- ✅ Good -->
<span class="error" aria-label="Required field">
  <span aria-hidden="true">*</span> Required field
</span>
```

## Form Accessibility

Create accessible forms:

```html
<form>
  <fieldset>
    <legend>Contact Information</legend>
    
    <div>
      <label for="name">Name</label>
      <input 
        id="name" 
        type="text" 
        required
        aria-describedby="name-help"
      />
      <small id="name-help">Enter your full name</small>
    </div>
    
    <div>
      <label for="email">Email</label>
      <input 
        id="email" 
        type="email" 
        required
        aria-invalid="false"
        aria-describedby="email-error"
      />
      <span id="email-error" role="alert"></span>
    </div>
  </fieldset>
  
  <button type="submit">Submit</button>
</form>
```

## Testing Accessibility

### Automated Tools

```bash
# Install axe DevTools
npm install @axe-core/react

# Lighthouse
npm install -g lighthouse
lighthouse http://localhost:3000 --view
```

### Manual Testing

1. **Keyboard navigation**: Tab through all interactive elements
2. **Screen reader**: Test with NVDA (Windows) or VoiceOver (Mac)
3. **Focus indicators**: Ensure visible focus states
4. **Headings**: Logical heading hierarchy
5. **Alt text**: All images have descriptive alt text

### React Testing

```typescript
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

test('modal closes on Escape key', async () => {
  const onClose = jest.fn();
  render(<Modal isOpen={true} onClose={onClose} />);
  
  await userEvent.keyboard('{Escape}');
  expect(onClose).toHaveBeenCalled();
});

test('button has accessible label', () => {
  render(<IconButton icon="close" ariaLabel="Close dialog" />);
  expect(screen.getByLabelText('Close dialog')).toBeInTheDocument();
});
```

## Common Patterns

### Skip Links

```html
<a href="#main-content" class="skip-link">
  Skip to main content
</a>

<main id="main-content">
  <!-- Content -->
</main>
```

### Loading States

```html
<button aria-busy="true" aria-label="Loading">
  <span aria-hidden="true">...</span>
  <span class="sr-only">Loading</span>
</button>
```

### Hidden Content

```css
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border-width: 0;
}
```

## WCAG Guidelines

- **Level A**: Basic accessibility
- **Level AA**: Recommended standard (legal requirement)
- **Level AAA**: Enhanced accessibility

Key principles:
1. **Perceivable**: Information must be presentable
2. **Operable**: Interface components must be usable
3. **Understandable**: Information must be clear
4. **Robust**: Content must be interpretable

Building accessible applications creates better experiences for everyone and demonstrates commitment to inclusive design.
