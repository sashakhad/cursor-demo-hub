Effective Git workflows are essential for productive team collaboration. Understanding different workflow strategies helps teams work together efficiently while maintaining code quality.

## Common Workflow Models

### Git Flow

Git Flow is a branching model designed around project releases.

**Branches:**
- `main`: Production-ready code
- `develop`: Integration branch for features
- `feature/*`: New features
- `release/*`: Prepare for production
- `hotfix/*`: Critical production fixes

```bash
# Feature development
git checkout -b feature/new-feature develop
git add .
git commit -m "Add new feature"
git checkout develop
git merge feature/new-feature

# Release preparation
git checkout -b release/1.0.0 develop
# Fix bugs, update version
git checkout main
git merge release/1.0.0
git tag 1.0.0
git checkout develop
git merge release/1.0.0
```

### GitHub Flow

Simpler workflow for continuous deployment:

```bash
# Create feature branch
git checkout -b feature/new-feature main

# Make changes and commit
git add .
git commit -m "Implement feature"

# Push and create PR
git push origin feature/new-feature

# After review, merge to main
git checkout main
git merge feature/new-feature
git push origin main
```

### GitLab Flow

Environment-based branching with upstream:

```bash
# Feature branch
git checkout -b feature/new-feature main

# After merge, push to staging
git checkout staging
git merge main
git push origin staging

# After testing, push to production
git checkout production
git merge staging
git push origin production
```

## Essential Git Commands

### Branching

```bash
# List branches
git branch -a

# Create and switch
git checkout -b feature/new-feature

# Rename branch
git branch -m old-name new-name

# Delete branch
git branch -d feature/old-feature
```

### Stashing

```bash
# Save work in progress
git stash

# List stashes
git stash list

# Apply stash
git stash apply

# Apply and remove
git stash pop

# Create branch from stash
git stash branch feature/new-branch
```

### Rebasing

```bash
# Interactive rebase
git rebase -i HEAD~3

# Rebase on main
git checkout feature/new-feature
git rebase main

# Rebase in progress
git rebase --continue
git rebase --abort
```

### Cherry-picking

```bash
# Apply specific commit
git cherry-pick <commit-hash>

# Cherry-pick range
git cherry-pick <start-hash>^..<end-hash>
```

## Commit Best Practices

### Write Good Commit Messages

```
Short summary (50 chars or less)

More detailed explanation if needed. Wrap at 72 characters.
Explain what and why, not how.

- Use imperative mood
- Reference issues
- Be specific
```

### Atomic Commits

```bash
# Stage specific files
git add src/components/Header.tsx

# Or use patches
git add -p

# Commit with message
git commit -m "Add header component with navigation"
```

## Pull Requests

### Good PR Practices

1. **Keep PRs small** - Easier to review
2. **Write clear titles** - Describe what it does
3. **Add descriptions** - Explain why and how
4. **Link issues** - Reference related issues
5. **Request review** - Tag relevant reviewers
6. **Fix feedback promptly** - Keep momentum

### PR Template

```markdown
## Description
Brief description of changes

## Related Issues
Closes #123

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change

## Testing
Describe tests performed

## Screenshots (if applicable)
```

## Advanced Techniques

### Interactive Rebase

```bash
# Squash commits
git rebase -i HEAD~3

# In editor:
pick 1a2b3c4 First commit
squash 2b3c4d5 Second commit
squash 3c4d5e6 Third commit
```

### Reflog Recovery

```bash
# View reflog
git reflog

# Recover lost commit
git checkout <hash>
git checkout -b recovery-branch
```

### Bisect

```bash
# Find bug-introducing commit
git bisect start
git bisect bad
git bisect good <known-good-commit>

# Mark current commit
git bisect good  # or bad

# End bisect
git bisect reset
```

## Useful Aliases

```bash
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.unstage 'reset HEAD --'
git config --global alias.last 'log -1 HEAD'
git config --global alias.visual '!gitk'
```

## Team Workflow Tips

1. **Regular pulls** - Keep local main updated
2. **Review PRs carefully** - Don't rush merges
3. **Use protected branches** - Enforce code quality
4. **Document conventions** - Share team standards
5. **Automate checks** - CI/CD for testing
6. **Communicate** - Discuss before major changes

Mastering Git workflows enables smooth collaboration and helps maintain high code quality across your projects.
